# WeMail 邮件列表页重设计稿

日期：2026-04-18

## 目标

为 `/mail/list` 重新设计一套以 **QA 快速拿验证码 / 链接** 为第一优先、同时兼顾研发调试邮件流程的邮件列表页面。

本次设计明确：

- 第一主用户：**QA / 测试同学**
- 第二辅助用户：**内部工具 / 研发同学**
- 第一任务：**快速看见验证码、链接和提取结果**
- 第二任务：在不跳出页面的前提下完成正文阅读、上下文确认和调试查看
- 页面整体仍应**像邮箱客户端**，而不是完全变成任务队列或运营后台
- 邮箱管理与发件能力需要保留，但应降为**辅助工具位**，不抢主信息流

## 设计结论

### 1. 整体方向

- 页面类型：**经典收件箱增强版**
- 方向组合：**A（经典收件箱骨架） + C（轻量摘要条）**
- 视觉原则：
  - 保留熟悉的邮箱阅读模型
  - 把验证码 / 链接提取结果提升为列表第一视觉焦点
  - 只引入轻量摘要，不做厚重 dashboard
  - 将邮箱管理、发件和调试动作收束到辅助区域

一句话概括：

> 这是一个“更适合 QA 拿码”的邮箱页面，而不是一个伪装成邮箱的任务看板。

### 2. 页面定位

邮件列表页是 **收件处理工作台**，不是邮箱资源后台，也不是运营总览页。

它首先服务这两类动作：

1. 快速扫读最近邮件，第一时间拿到验证码 / 登录链接 / 提取结果
2. 点进详情确认正文、来源、提取上下文和调试信息

因此它的设计重点不是“展示所有功能”，而是：

- 让用户进入页面后立即知道当前正在看哪个邮箱
- 让用户在消息列表里优先扫到“提取到什么”
- 让用户在详情区中先完成任务，再看完整正文

## 页面结构

### 1. 总体结构

页面采用 **顶部轻量摘要条 + 三栏主工作区** 结构。

#### 第一段：顶部轻量摘要 / 工具条

该区域只承担“快速建立上下文”和“放置高频动作”的职责，不承担完整运营看板职责。

包含：

- 当前邮箱信息
  - 邮箱名称
  - 邮箱地址
  - 当前是否默认 / 是否活跃
- 聚合摘要
  - 待提取数量
  - 未读数量
  - 附件数量
- 高频动作
  - 切换邮箱
  - 新建邮箱
  - 发送测试邮件
- 轻量筛选入口
  - 仅看未读
  - 仅看验证码
  - 仅看链接

设计要求：

- 高度紧凑
- 一屏内不抢主视线
- 作用是“抬头即懂”，而不是“停下来分析数据”

#### 第二段：左侧邮箱辅助栏

左栏用于邮箱切换与基础上下文确认，定位为辅助导航。

每个邮箱项只展示：

- 名称
- 地址
- 消息数量

不在该区域堆叠复杂操作。第一版不在左栏直接放：

- 删除
- 高风险编辑
- 多层管理菜单

如果需要更多邮箱管理动作，应通过顶部入口、次级菜单或后续独立页面承载。

#### 第三段：中间消息列表

这是页面的主工作区，承担第一视觉重点。

消息列表遵循固定信息优先级：

1. **提取结果**
   - 验证码
   - 登录链接
   - Magic link
   - 未提取状态
2. 发件人
3. 时间
4. 主题 / 主题补充信息

也就是说，列表行不是传统“主题优先”的收件箱，而是“提取结果优先”的收件箱。

#### 第四段：右侧消息详情区

详情区负责完成“确认与操作”。

详情区顶部必须优先展示：

- 超醒目的验证码 / 提取结果
- 快捷操作：
  - 复制验证码
  - 打开原始邮件
  - 查看提取 JSON

其后再展示：

- 正文预览
- 发件上下文
- 提取信息详情
- 附件 / 调试信息 / 辅助信息

原则：

- **先完成任务，再阅读正文**
- 详情区仍然像邮件阅读面板，不做成日志控制台

## 列表项设计

### 1. 列表项核心规则

每一条消息卡统一采用以下顺序：

1. 提取结果区
2. 发件人 / 时间
3. 主题 / 摘要
4. 状态标签（如有）

### 2. 提取结果显示策略

#### 验证码邮件

如果邮件识别出验证码：

- 使用大号 code chip 展示
- 数字需要有足够高的对比度和尺寸
- 视觉上明显强于发件人和主题

示例：

- `482913`
- `761228`

#### 链接邮件

如果识别出登录链接或 magic link：

- 使用高识别度标签展示
- 文案优先使用：
  - `LOGIN LINK`
  - `MAGIC LINK`
- 不直接在列表里堆放长 URL

#### 未识别邮件

如果未提取到有效验证码或链接：

- 统一显示 `未提取`
- 仍展示发件人与主题
- 保持视觉降级，避免干扰高价值消息

### 3. 状态标签

列表中允许附加轻量状态标签，但不得压过提取结果。

第一版允许使用：

- 未读
- 已提取验证码
- 已识别链接
- 有附件
- 提取失败（如后续需要）

## 详情区设计

### 1. 顶部任务区

详情区顶部是页面的第二视觉中心。

包含：

- 当前消息提取结果的大号展示
- 一组任务型快捷操作

优先动作：

- 复制验证码
- 打开原始邮件
- 查看提取 JSON

后续可追加但不应抢权重：

- 查看附件
- 查看完整 headers
- 调试提取链路

### 2. 正文阅读区

正文区应保持典型邮件阅读体验：

- 标题与基础元信息清晰
- 正文可直接阅读
- 不被过多调试字段打断

### 3. 上下文 / 调试区

右侧详情中的次级模块可展示：

- 发件时间
- 发件人地址
- 提取类型
- 提取结果值
- 置信度
- SMTP / inbound 处理来源
- 附件信息

这些模块应采用次级卡片展示，不抢正文和验证码区域的主次。

## 邮箱管理与发件策略

### 1. 邮箱管理

邮箱管理仍保留在此页，但只保留高频入口：

- 切换邮箱
- 新建邮箱

更复杂的邮箱管理动作不在本次重做里增强。

原因：

- 用户在该页面的第一任务不是管理邮箱资源
- 当前目标是把注意力集中在收件处理与提取结果

### 2. 发件与调试

当前发件能力不再与主阅读区并列占据大面积空间。

第一版明确采用以下方案：

- `发送测试邮件` 通过顶部动作打开右侧抽屉
- 调试信息保留在详情区次级卡片中
- 抽屉默认关闭，不占主阅读区宽度

要求：

- 默认不打断“扫列表拿码”的主流程
- 需要时仍可快速进入

本次设计不要求把发件独立为新页面。

## 架构与组件拆分建议

### 1. 页面路由

- 保持 `/mail/list` 不变
- 保持当前工作台导航结构不变
- 不新增一级导航

### 2. 前端组件拆分

建议在现有 `InboxPage` 基础上重组，而不是推翻路由层。

第一版建议拆分为：

- `InboxSummaryBar`
  - 顶部摘要与高频动作
- `MailboxNavPanel`
  - 左侧邮箱辅助栏
- `InboxMessageListPanel`
  - 中间消息列表
- `InboxMessageRow`
  - 单条消息展示
- `InboxDetailPanel`
  - 右侧详情主区
- `InboxDetailMetaCards`
  - 发件上下文 / 提取信息 / 调试卡片
- `InboxAuxPanel` 或 `InboxComposerDrawer`
  - 发件 / 辅助操作区域

已有组件若可复用，应优先调整责任边界而不是新增平行实现。

### 3. 数据流

第一版不增加新接口，基于现有数据流重组页面。

现有数据关系保持：

- `selectedMailboxId` 驱动消息列表刷新
- `selectedMessageId` 驱动详情区展示
- `outboundHistory` 继续保留，但默认不占主视觉
- `selectedMessage` 仍通过现有 view-model 转换为详情展示数据

也就是说：

- **主要变化是信息编排，不是数据模型改造**
- 后端接口与 hook 逻辑应尽量保持不变

## 状态与异常处理

### 1. 空状态

必须覆盖以下空状态：

#### 无邮箱

- 左栏显示无邮箱引导
- 顶部保留“新建邮箱”入口
- 主区域不给出空白页面

#### 有邮箱但无消息

- 列表区显示明确空状态
- 文案应强调“新邮件到达后会显示在这里”
- 详情区保持任务引导，而不是报错感

#### 有消息但未选择详情

- 详情区显示默认提示
- 提示用户选择一封消息查看验证码、正文和提取结果

### 2. 异常状态

第一版需要对以下异常有明确展示策略：

- 提取失败
- 邮件正文为空
- 附件缺失
- 发件动作不可用

原则：

- 错误信息就近展示
- 不使用笼统失败提示替代具体状态
- 错误卡片保持次级，不破坏整体阅读结构

## 响应式策略

### 1. 桌面端

优先采用三栏结构：

- 左：邮箱辅助栏
- 中：消息列表
- 右：消息详情

### 2. 中等屏幕

中屏允许退化为：

- 顶部摘要条保持
- 左侧邮箱栏变窄
- 详情区保持主阅读权重

### 3. 小屏 / 窄屏

第一版不追求完整还原桌面效率，但需保证：

- 顶部摘要条可折行
- 邮箱切换与列表阅读仍可使用
- 详情区可单独进入

原则是“可用”，不是“完整桌面复刻”。

## 视觉风格

- 主基调：干净、克制、工具化
- 风格关键词：
  - 邮箱客户端
  - QA 工具台
  - 提取结果优先
- 避免：
  - 大面积运营看板化
  - 过多统计卡
  - 让页面看起来像任务系统
- 强化：
  - 验证码的可扫读性
  - 状态标签的层次感
  - 正文阅读的稳定感

## 实现边界

本次设计包含：

- `/mail/list` 的页面信息架构重排
- 顶部轻量摘要条
- 左侧邮箱辅助栏
- 提取结果优先的消息列表
- 任务优先的详情区
- 发件 / 辅助能力降级为次级区域

本次设计不包含：

- 新后端接口
- 新消息数据结构
- 邮箱复杂管理动作增强
- 发件能力独立成新页面
- 无收件人邮件页重做
- 邮件设置页重做

## 验证建议

至少补以下验证：

### 1. 组件 / 页面测试

- `/mail/list` 不再渲染旧的四块并列结构
- 顶部摘要条出现
- 左侧邮箱辅助栏出现
- 列表项优先展示提取结果
- 详情区顶部优先展示验证码 / 链接提取结果
- 发件区域不再与主阅读区同权重并列

### 2. 交互测试

- 切换邮箱后消息列表正确更新
- 选择消息后详情区正确更新
- 无提取结果消息仍可查看正文
- 空列表与空详情提示正常

### 3. smoke / e2e

- 进入 `/mail/list` 可见新布局骨架
- 导航与页面标题保持正常
- 基本收件浏览流程不被破坏

## 结论

邮件列表页第一版应实现为：

- **像邮箱客户端的收件处理工作台**
- **顶部带一条轻量摘要 / 工具条**
- **消息列表以验证码 / 链接提取结果为第一视觉重点**
- **详情区优先服务复制、确认和调试**
- **邮箱管理与发件能力保留，但降为辅助位**
