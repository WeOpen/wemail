# WeMail 邮箱账号列表与全局设置设计稿

日期：2026-04-17

## 目标

为 `/accounts/list` 与 `/accounts/settings` 重新设计一套围绕**邮箱账号管理**的后台页面。

本次设计明确：

- “账号”指的是**邮箱账号**
- 账号列表页第一优先目标是**批量管理账号**
- 批量管理的最高优先动作是：
  - 批量启用 / 停用
  - 批量删除
- 删除语义分层：
  - 支持软删除 / 归档
  - 支持彻底删除
  - 默认流程优先引导到软删除 / 归档
- 账号设置页定位为**所有邮箱账号的全局设置中心**

## 设计结论

### 1. 整体方向

- 页面类型：后台控制台页
- 视觉方向：**批量操作控制台型**
- 交互原则：
  - 不保留大 Hero
  - 优先把空间让给表格、筛选、批量操作和规则配置
  - 高风险动作与普通动作分层

### 2. 页面关系

本次重做的页面关系如下：

1. `/accounts/list`
   - 面向“执行”
   - 用于筛选、搜索、多选、批量启停、批量软删除 / 归档、批量彻底删除

2. `/accounts/settings`
   - 面向“定规则”
   - 用于配置默认创建规则、生命周期规则、批量操作保护

本次**不包含**：

- 单个邮箱账号详情页
- 单个邮箱账号编辑页
- `/accounts/create` 的重做
- 接入真实后端接口

## 账号列表页

### 1. 页面定位

账号列表页是**邮箱账号批量管理控制台**，不是资源概览页，也不是单账号详情页。

第一目标是让管理员能够快速完成以下任务：

- 定位一批账号
- 判断这些账号是否需要处理
- 对它们执行批量启用 / 停用 / 删除

### 2. 页面结构

账号列表页采用三段式结构：

#### 第一段：顶部紧凑工具栏

包含：

- 页面标题：`账号列表`
- 搜索框：支持按以下字段搜索
  - `ID`
  - `地址`
  - `标签`
  - `创建人`
- 筛选器：
  - 状态
  - 标签
  - 创建人
  - 最近活跃时间
- 快捷筛选：
  - 仅看异常
  - 仅看长期不活跃
- 右侧快捷动作：
  - `创建账号`
  - `导出`
  - `刷新`

顶部工具栏应保持紧凑，不使用大说明区。

#### 第二段：账号主表格

页面核心是表格，列顺序固定为：

1. 多选框
2. ID
3. 地址
4. 创建时间
5. 标签
6. 状态
7. 创建人
8. 最近活跃
9. 邮件数量
10. 发件数量
11. 行级操作

字段表现规则：

- **标签**
  - 使用 tag / chip 展示
  - 支持多标签
- **状态**
  - 使用清晰的状态 badge
  - 第一版统一为：
    - 启用
    - 停用
    - 已归档
    - 已软删除
- **邮件数量 / 发件数量**
  - 直接显示数值
  - 允许后续增加排序
- **最近活跃**
  - 使用清晰的时间文本
  - 支持后续按阈值标记“不活跃”

行级操作只保留轻量入口：

- 启用 / 停用
- 归档
- 更多操作

#### 第三段：批量操作条

当用户勾选至少一个账号后，出现批量操作条。

批量操作条显示：

- 已选账号数量
- 批量启用
- 批量停用
- 批量归档
- 更多操作

“更多操作”中放置：

- 批量软删除
- 批量彻底删除

批量操作条可以采用顶部固定条或底部悬浮条，但必须满足：

- 不选中时完全隐藏
- 选中后反馈强烈
- 危险操作不能和普通操作同权重展示

### 3. 默认排序与信息节奏

账号列表默认按**最近活跃倒序**展示。

允许后续切换的排序包括：

- 创建时间
- 邮件数量
- 发件数量

第一屏应优先帮助管理员看到：

- 最近活跃账号
- 长期不活跃账号
- 停用账号
- 高量级账号

也就是说，这一页的目标不是“浏览资源”，而是“快速处理一批账号”。

### 4. 批量操作交互

#### 普通批量动作

包括：

- 批量启用
- 批量停用
- 批量归档

交互流程统一为：

1. 用户勾选账号
2. 点击批量动作
3. 打开确认框
4. 明确展示影响账号数量
5. 用户确认执行
6. 页面返回 toast 与表格更新

#### 高风险批量动作

包括：

- 批量软删除
- 批量彻底删除

危险动作规则：

- 不出现在主操作按钮组第一层
- 必须进入 `danger` 层级
- 必须展示操作影响说明

其中“批量彻底删除”流程要求最高，确认框中必须展示：

- 本次影响账号数
- 是否存在仍有邮件数据的账号
- 操作不可恢复

并要求二次确认，建议使用输入确认词，例如：

- `DELETE 12 ACCOUNTS`

### 5. 空状态与异常状态

列表页必须处理以下状态：

- 无账号数据
  - 主要引导：`创建账号`
- 筛选无结果
  - 主要引导：清空筛选
- 批量操作全部失败
  - 返回失败原因摘要
- 批量操作部分成功
  - 返回“成功 X 个 / 失败 Y 个”

不允许只给出笼统失败提示。

## 账号设置页

### 1. 页面定位

账号设置页是**所有邮箱账号的全局设置中心**。

这页不负责编辑单个账号，而是定义：

- 新账号创建时默认采用什么规则
- 账号生命周期如何流转
- 批量操作如何被保护

一句话概括：

- 账号列表页负责**执行**
- 账号设置页负责**定规则**

### 2. 页面结构

采用 **左主内容 + 右侧摘要栏** 结构。

#### 左侧主内容

按三个模块顺序排列：

1. 默认创建规则
2. 生命周期规则
3. 批量操作保护

每个模块独立成卡片，并支持**独立保存**。

#### 右侧摘要栏

摘要栏常驻显示：

- 当前默认创建状态
- 当前归档 / 软删保留策略
- 当前危险操作保护级别
- 最近一次规则更新时间
- 本页规则影响范围说明

摘要栏的作用是让页面更像策略控制台，而不是纯表单堆叠。

### 3. 模块一：默认创建规则

这一模块用于保证新账号一进入系统就符合统一标准。

建议包含：

#### 默认标签规则

- 是否自动附加默认标签
- 默认标签值
- 创建时是否允许覆盖默认标签

#### 命名 / 标识规则

- 地址命名约定说明
- 标签命名模板
- 是否自动生成显示名

#### 默认状态

- 创建后默认启用
- 创建后默认停用
- 预留待审核状态入口

#### 创建元信息

- 是否强制记录创建人
- 是否要求备注
- 是否自动记录创建来源

### 4. 模块二：生命周期规则

这一模块用于定义邮箱账号从启用到删除的标准路径。

#### 账号状态流转

页面中应明确展示状态流：

- 启用
- 停用
- 已归档
- 已软删除
- 彻底删除

推荐使用可视化状态流说明，而不是纯文字段落。

#### 不活跃处理规则

包含：

- 多久未活跃视为不活跃
- 不活跃后的建议动作：
  - 仅标记
  - 自动停用
  - 自动归档

#### 软删除策略

包含：

- 软删除后的保留期
- 是否允许恢复
- 恢复后回到什么状态

#### 彻底删除策略

包含：

- 是否允许彻底删除
- 是否必须先软删除再彻底删除
- 软删除后需等待多久才能彻底删除
- 是否始终展示不可恢复警告

### 5. 模块三：批量操作保护

这一模块是账号批量管理设计中的关键保护层。

#### 普通批量操作保护

针对：

- 批量启用
- 批量停用
- 批量归档

配置项包括：

- 是否显示确认弹窗
- 单次操作上限
- 是否显示影响数量预览

#### 高风险批量操作保护

针对：

- 批量软删除
- 批量彻底删除

配置项包括：

- 是否必须二次确认
- 是否要求输入确认词
- 是否需要更高权限角色执行
- 单次彻底删除最大数量限制

#### 审计与记录

配置项包括：

- 是否记录批量操作日志
- 日志至少记录：
  - 操作人
  - 操作时间
  - 动作类型
  - 影响账号数
- 是否允许导出操作记录

### 6. 保存与生效方式

账号设置页不采用自动保存。

统一规则：

- 每个模块独立保存
- 有改动时显示未保存状态
- 保存成功后给明确反馈
- 危险策略变更需要额外确认

需要额外确认的典型场景：

- 开启“允许彻底删除”
- 缩短软删除保留期
- 降低高风险动作确认级别

## 风险控制原则

本次设计明确采用以下风险控制原则：

1. 普通批量操作追求效率
2. 高风险批量操作追求可解释、可确认、可审计
3. 彻底删除不与普通操作并列
4. 列表页只负责执行，不承载全局策略判断
5. 设置页只负责规则，不承载单账号编辑

## 测试重点

实现阶段应优先覆盖以下测试：

### 1. 账号列表页

- 搜索与筛选联动
- 多选状态与批量操作条出现 / 消失
- 批量启用 / 停用 / 归档确认流程
- 批量软删除与彻底删除分层展示
- 部分成功 / 部分失败结果反馈

### 2. 账号设置页

- 默认创建规则保存
- 生命周期规则保存
- 批量操作保护保存
- 危险配置变更二次确认
- 模块独立保存，不互相污染

### 3. 风险与权限

- 成员账号不可见危险全局配置
- 危险批量动作遵守保护策略
- 彻底删除前确认词校验有效

## 实施边界

第一版实现建议：

- 先完成静态页面结构与 mock 数据
- 先把列表页的批量管理体验做完整
- 设置页先完成结构与交互骨架
- 真实批量策略与后端规则在后续接入

这样可以先把信息架构、交互层级和危险动作边界稳定下来，再接真实能力。
