# WeMail 发件箱与邮件设置重设计稿

日期：2026-04-19

## 目标

在邮件工作台中完成以下结构调整与页面重做：

1. **移除“无收件人邮件”独立页面**
2. 重新设计 `/mail/outbound`，使其成为 **发件记录中心**
3. 重新设计 `/mail/settings`，使其成为以 **发件规则 + 通知 / 路由** 为主的整合设置页

本次设计明确：

- 发件箱方向：**D（可发送 + 可查历史）偏 B（记录中心）**
- 邮件设置方向：**D（整合页）偏 A+C（发件规则 + 通知 / 路由）**
- “无收件人邮件”不再保留为独立二级入口
- “无收件人邮件 / 无匹配邮件 / 异常邮件”能力并入 **发件箱**

## 设计结论

### 1. 邮件分区的新结构

邮件区二级导航收敛为三个入口：

1. `/mail/list` — 邮件列表
2. `/mail/outbound` — 发件箱
3. `/mail/settings` — 邮件设置

新的职责分工如下：

- **邮件列表**：负责收件阅读、提取结果查看、正文确认
- **发件箱**：负责发件历史、状态检索、失败记录、异常 / 无匹配邮件处理
- **邮件设置**：负责发件规则、通知策略、路由策略与邮件工作台偏好

### 2. 旧入口处理

#### 移除的入口

从邮件二级导航中删除：

- `无收件人邮件`

#### 旧路由处理

旧路由 `/mail/unassigned` 不再承载独立真实页面。

它应：

- 自动跳转到 `/mail/outbound`
- 并默认落在 **“异常 / 无匹配”** 视图

也就是说：

- 原本“无收件人邮件页面”的概念消失
- 用户仍然可以处理这类邮件，但入口改到发件箱中完成

## 发件箱页

### 1. 页面定位

新版发件箱页不是“写邮件页”，而是 **发件记录中心**。

它的第一目标是让用户能够快速完成以下任务：

- 检索最近发送记录
- 判断发送状态
- 查看失败原因
- 定位异常 / 无匹配邮件
- 对记录执行重发或查看 payload 等操作

因此页面主视角应当是：

- **历史记录与异常队列**

而不是：

- 永久展开的发送表单

### 2. 页面结构

发件箱采用 **顶部控制条 + 左主记录列表 + 右详情区** 结构。

#### 第一段：顶部控制条

顶部控制条包含：

- 页面标题：`发件箱`
- 搜索框：支持按以下字段搜索
  - 收件人
  - 主题
  - 发件结果
- 状态筛选：
  - 全部
  - 已发送
  - 失败
  - 异常 / 无匹配
- 高频动作：
  - `新建发送`
  - `刷新`

设计要求：

- 保持紧凑
- 把动作做成入口，不让“新建发送”压过历史记录区域

#### 第二段：左侧主记录列表

列表按时间倒序排列，以发件记录为主。

每条记录建议展示：

- 状态标签
  - 已发送
  - 失败
  - 异常 / 无匹配
- 收件人
- 主题
- 时间
- 一行结果摘要

其中：

- **异常 / 无匹配** 记录需要有更高识别度
- 这类记录就是原“无收件人邮件”能力的新承载形式

#### 第三段：右侧详情区

点中某条记录后，右侧展示：

- 基本信息
- 发件结果 / 错误原因
- 对应 payload / 调试摘要
- 操作按钮：
  - 重发
  - 复制 payload
  - 查看原始详情

如果当前选中的是“异常 / 无匹配”项，则详情区需展示：

- 原始收件信息
- 未命中邮箱 / 路由的原因
- 后续处理建议

### 3. “新建发送”的处理方式

`新建发送` 需要保留，但不应再作为页面主结构的一部分永久占位。

第一版建议：

- 使用抽屉
- 或对话框
- 或切换右侧详情区到“新建发送模式”

但必须满足：

- 默认不占主区空间
- 不破坏发件箱的记录中心定位

本次设计推荐：

- 优先使用 **抽屉或对话框**

原因：

- 用户目标已经明确偏向“看记录、查异常”
- 永久展开表单会稀释记录中心的主次

### 4. 异常 / 无匹配视图

原“无收件人邮件”能力在发件箱中变成一个重点视图。

展示对象包括：

- 无匹配邮箱的异常项
- 路由失败的异常项
- 需要人工确认的发件异常项（如后续扩展）

在控制条和筛选结构上，应明确提供：

- `异常 / 无匹配`

让用户无需再理解“为什么有单独页面”，而是把它视作发件箱中的一个特殊记录分组。

## 邮件设置页

### 1. 页面定位

新版邮件设置页是 **整合设置页**，但主重点明确放在：

1. 发件规则
2. 通知与路由

其他偏好设置只作为次级内容。

一句话概括：

- 这页是 **规则中心 / 控制中心**
- 不是普通偏好设置页

### 2. 页面结构

邮件设置页采用 **左主内容 + 右摘要栏** 结构。

#### 左侧主内容

按三个模块顺序排列：

##### 模块 A：发件规则

这是第一主模块，应位于页面最上方。

建议包含：

- 默认发件身份 / 发件邮箱
- 默认签名
- 失败后是否允许重试
- 重试次数 / 重试策略
- 失败记录保留方式

这一块负责定义：

- 系统“怎么发”

因此权重最高。

##### 模块 B：通知与路由

这是第二主模块。

建议包含：

- Webhook 通知
- Telegram 通知
- 失败告警
- 异常 / 无匹配邮件是否通知
- 不同事件是否分渠道推送

这一块负责定义：

- 系统“发出问题后怎么通知、怎么流转”

##### 模块 C：工作台行为偏好

这是第三层内容，放在页面后半部分。

建议包含：

- 默认进入哪个邮件子页面
- 发件箱默认筛选
- 是否默认展开异常记录
- 邮件列表阅读偏好（如后续接入）

这一块有价值，但不应抢前两块的主次。

#### 右侧摘要栏

右侧只做轻量摘要，不承载复杂表单。

建议展示：

- 当前默认发件身份
- 当前失败告警是否开启
- 当前 Webhook / Telegram 状态
- 当前异常邮件处理策略

作用是：

- 让用户不滚动也能快速理解当前邮件系统策略

### 3. 页面气质

邮件设置页的语气和布局都应偏：

- 规则中心
- 控制中心

而不是：

- 普通偏好页
- 说明文档页

所以文案和层级必须明确表达：

- 哪些是默认行为
- 哪些是失败策略
- 哪些是通知策略
- 哪些只是工作台偏好

## 两页之间的关系

新版发件箱与邮件设置需要形成清晰分工：

- **发件箱**：看结果、查历史、处理异常
- **邮件设置**：定规则、定路由、定告警

这样用户在心智上会非常清楚：

- 想看“发生了什么” → 去发件箱
- 想改“以后怎么做” → 去邮件设置

## 路由与导航调整建议

### 1. `workspaceShell.ts`

邮件分区子导航调整为：

- `邮件列表`
- `发件箱`
- `邮件设置`

删除：

- `无收件人邮件`

### 2. `AppRoutes.tsx`

调整原则：

- 删除独立 `mailUnassignedPage` 的存在感
- `/mail/unassigned` 改为跳转到 `/mail/outbound`
- 并保留“异常 / 无匹配”视图的承接逻辑
- `/mail/outbound` 与 `/mail/settings` 不再使用简单占位页，而是替换成真实页面

## 实现边界

本次设计包含：

- 移除邮件区二级导航中的“无收件人邮件”
- `/mail/unassigned` 跳转到 `/mail/outbound`
- `/mail/outbound` 重做为发件记录中心
- `/mail/settings` 重做为规则中心型整合设置页
- 异常 / 无匹配邮件并入发件箱

本次设计不包含：

- 新后端接口
- 真正的批量群发系统
- 复杂发信审批流
- 邮件区新的一级导航拆分
- 收件页再次大改

## 验证建议

至少补以下验证：

### 1. 导航与路由

- 顶部邮件二级导航不再出现“无收件人邮件”
- `/mail/unassigned` 自动跳转到 `/mail/outbound`
- 新的发件箱和邮件设置路由可正常访问

### 2. 发件箱页

- 出现发件记录中心型布局
- 出现状态筛选
- 出现异常 / 无匹配视图或入口
- 出现新建发送入口但不作为永久主区
- 记录详情区可查看发送结果 / 错误 / payload 信息

### 3. 邮件设置页

- 第一屏先出现发件规则模块
- 第二重点出现通知与路由模块
- 右侧摘要栏出现当前策略概览
- 页面不再只是简单跳转卡片占位

### 4. smoke / e2e

- 已登录成员可访问新版发件箱
- 已登录成员可访问新版邮件设置页
- 邮件二级导航和标题保持正确
- 旧的“无收件人邮件”导航断言更新

## 结论

邮件区本轮重做应实现为：

- **只保留：邮件列表 / 发件箱 / 邮件设置 三个二级入口**
- **无收件人邮件不再独立存在，而是并入发件箱的异常 / 无匹配视图**
- **发件箱成为偏历史中心的发件记录页**
- **邮件设置成为偏发件规则 + 通知 / 路由的整合规则中心**
