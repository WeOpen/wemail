# 分层架构说明

## 目标

本项目采用“前后端分别分层、共享契约统一复用”的结构，避免：

- 单文件膨胀
- 表现层与业务逻辑耦合
- 路由层直接持久化
- 工具函数散落
- 目录职责不清

---

## 顶层结构

```text
apps/
  web/       # 前端应用（React + Vite）
  worker/    # 后端应用（Cloudflare Worker + Hono）
packages/
  shared/    # 前后端共享类型、常量、纯函数
scripts/     # 仓库级脚本
.docs/       # 不存在；统一使用 docs/
docs/        # 项目文档与规范
```

---

## 前端分层

```text
apps/web/src/
  app/       # 应用装配、入口、路由、全局状态编排
  pages/     # 页面级组合层，只做页面结构组织
  features/  # 业务特性（可逐步演进）
  shared/    # 通用 api、样式、hooks、ui、utils
  test/      # 测试初始化、测试工具、页面级测试
```

### 前端分层规则

1. `app/` 负责启动与装配，不承载可复用业务组件。
2. `pages/` 负责组合多个特性区块，不直接写底层 fetch 工具。
3. `features/` 放明确业务能力，例如 auth / inbox / settings / admin。
4. `shared/` 只放无业务归属的共用能力。
5. 页面不得横向依赖其它页面。

---

## 后端分层

```text
apps/worker/src/
  app/              # 应用装配层、HTTP 入口编排、运行时流程
  core/             # 类型契约、绑定定义、核心上下文契约
  infrastructure/   # D1/R2/外部集成/持久化实现
  shared/           # 通用安全、邮件解析、纯工具
```

### 后端分层规则

1. `app/` 负责路由装配与请求流程，不应承载底层存储细节。
2. `core/` 只定义接口、类型、绑定、上下文，不写具体基础设施实现。
3. `infrastructure/` 负责数据库、对象存储、外部服务接入。
4. `shared/` 放可复用的纯逻辑或通用辅助逻辑。
5. 基础设施层不能反向依赖页面或测试专用代码。

---

## shared 包定位

`packages/shared` 只允许存放：

- 共享类型
- 共享常量
- 纯函数
- 前后端都能安全使用的无副作用工具

不允许存放：

- Cloudflare 专属绑定访问
- 浏览器 DOM 操作
- 数据库存取逻辑
- 运行时环境分支过强的代码

---

## README 约束

每个关键目录都必须有 `README.md`，说明：

- 目录职责
- 放什么
- 不放什么
- 常见误用
- 与相邻层的边界

---

## 演进原则

- 先保证“职责清楚”，再追求“层数完整”
- 小项目不做过度抽象，但目录边界必须先立住
- 新增目录前先判断是否真的形成稳定职责
- 能放在现有层就不要制造新层
